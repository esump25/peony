PEONY: Personalized Wellness Curator
Peony is a wellness-focused web application that helps users discover local services and retail experiences based on their aesthetic preferences. The project uses a custom personality quiz to categorize users into specific "vibes," such as Glow Girl, Zen, or Power, and then uses geolocation to suggest actual businesses nearby that match that category.

To use the application, a user starts the quiz and answers ten lifestyle questions. Once the quiz is complete, the app calculates a "winner" based on the scores and presents an itinerary. The user can then click on the suggested service or retail slots to find real local businesses. After visiting or selecting a spot, users can leave reviews that are shared with the community in a live feed. The app also features a calendar integration that allows users to save their entire curated day to Google or Apple/Outlook calendars.

I am most proud of the database and the calendar integration. I decided to include a database through render's SQL feature because the database homework was one of the most interesting to me. It functions to save customer reviews. The calendar integration is also a highlight, as it programmatically generates .ics files and Google Calendar links that include the specific names, addresses, and tips generated by the quiz, creating a seamless bridge between a digital recommendation and a real-world schedule.

To run this project locally, you must first clone the repository and ensure you have a local server environment (like Live Server in VS Code) to open the index.html file. The frontend communicates with a backend service, so you will need to ensure the RENDER_URL variable at the top of the script.js file is pointing to the correct live or local backend endpoint. Because the app relies on geolocation, you must grant the browser permission to access your location when prompted.

Secrets in this project, specifically the Google Places API key and database credentials, are handled entirely on the backend server. No sensitive API keys are stored in the frontend script.js file or the public repository. The backend service uses environment variables to securely store these credentials, and the frontend simply makes requests to the backend proxy. This ensures that the Google API key is never exposed to the client or anyone browsing the source code on GitHub.
